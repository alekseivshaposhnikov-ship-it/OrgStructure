# Функциональное описание приложения «Org Structure Viewer»

## 1. Назначение

Приложение предназначено для визуализации и сценарного моделирования организационной структуры холдинга LEGENDA.

Пользователь может:

* просматривать дерево подразделений;
* выбирать подразделение в левом дереве;
* видеть выбранное подразделение и все его подчиненные элементы в организационной диаграмме;
* отображать подразделения, сотрудников и вакансии;
* скрывать и показывать вакансии;
* анализировать численность сотрудников и вакансий;
* открывать детальную карточку сотрудника;
* переключать дизайн карточек подразделений;
* экспортировать текущую диаграмму в PDF;
* создавать сценарий моделирования организационной структуры;
* анализировать текущую структуру AS IS;
* формировать целевую структуру TO BE;
* добавлять подразделения;
* добавлять сотрудников;
* добавлять вакансии;
* редактировать подразделения;
* редактировать сотрудников;
* редактировать вакансии;
* удалять подразделения;
* удалять сотрудников;
* удалять вакансии;
* перемещать подразделения;
* перемещать сотрудников между подразделениями;
* перемещать вакансии между подразделениями;
* видеть список изменений сценария;
* сравнивать текущую и целевую структуру;
* сбрасывать сценарий моделирования;
* видеть проекты сотрудников и вакансий в карточках организационной диаграммы;
* видеть административного ассистента рядом с директором/руководителем выбранной верхней структуры, а не в нижнем списке сотрудников.

---

## 2. Источник данных

Данные загружаются из API:

```http
GET /api/getDepartmentVacancy
```

В режиме локальной разработки запрос `/api` проксируется через Vite на корпоративный API.

Если загрузка данных не удалась, отображается сообщение об ошибке.

После успешной загрузки данные преобразуются во внутреннюю структуру приложения.

На основании полученной структуры создаются две версии данных:

* текущая структура AS IS;
* рабочая структура сценария TO BE.

Текущая структура AS IS остается неизменной.

Все сценарные изменения применяются только к рабочей структуре TO BE.

На текущем этапе изменения хранятся локально в памяти браузера и не сохраняются в базу данных.

---

## 3. Корневой узел

Приложение создает виртуальный корневой узел:

```text
Холдинг LEGENDA
```

Узел не приходит из API и формируется на стороне приложения.

Для корневого узла задаются:

* название холдинга;
* руководитель: Селиванов Василий Геннадьевич;
* должность: Генеральный директор;
* суммарная численность сотрудников;
* суммарное количество вакансий;
* общая численность с учетом вакансий.

Особенности:

* узел создается динамически при построении дерева;
* не сохраняется в API;
* агрегирует показатели всех корневых подразделений;
* используется как верхний уровень для текущей структуры;
* используется как верхний уровень для целевой структуры;
* используется как верхний уровень для режима отображения изменений.

При переключении между режимами отображения виртуальный корневой узел создается заново на основании соответствующей версии дерева.

---

## 4. Левая панель навигации

В левой панели отображается дерево подразделений.

В дереве отображаются:

* виртуальный корневой узел;
* подразделения;
* вложенные подразделения.

В дереве не отображаются:

* сотрудники;
* вакансии;
* руководители;
* показатели численности.

Пользователь может:

* раскрывать и сворачивать подразделения по стрелке;
* раскрывать и сворачивать подразделения двойным кликом по названию;
* выбирать подразделение кликом по названию.

Выбранное подразделение подсвечивается.

После выбора подразделения организационная диаграмма перестраивается и отображает выбранное подразделение как корень текущей визуализации.

Если пользователь выбрал подразделение и затем внес сценарное изменение, выбранное подразделение сохраняется.

Например:

1. пользователь выбрал отдел в левом дереве;
2. добавил вакансию в этом отделе;
3. диаграмма продолжает отображать структуру выбранного отдела;
4. приложение не сбрасывает отображение на корневой узел холдинга.

Если выбранное подразделение было удалено или перемещено так, что его невозможно найти в текущем режиме отображения, приложение возвращается к виртуальному корневому узлу.

---

## 5. Организационная диаграмма

В правой части отображается организационная диаграмма выбранного узла.

Диаграмма включает:

* выбранное подразделение;
* все дочерние подразделения;
* сотрудников подразделений;
* вакансии, если включен режим показа вакансий;
* карточку административного ассистента рядом с карточкой директора/руководителя выбранной верхней структуры, если такой сотрудник найден в текущей ветке;
* проекты сотрудников и вакансий внутри соответствующих карточек диаграммы.

После выбора подразделения диаграмма перестраивается и отображает выбранное подразделение как корень текущей визуализации.

После построения выполняется автоматическое масштабирование диаграммы.

В диаграмме отображаются карточки:

* подразделений;
* сотрудников;
* вакансий;
* административного ассистента, если сотрудник с такой должностью найден в отображаемой ветке.

Административный ассистент не должен отображаться в нижнем общем списке сотрудников как обычная карточка, если он уже выведен рядом с директором/руководителем верхнего подразделения выбранной визуализации.

В режиме сценарного моделирования на карточках отображается контекстное меню действий.

Контекстное меню открывается по кнопке:

```text
⋮
```

В режиме текущей структуры AS IS контекстное меню не отображается, чтобы пользователь не мог менять исходные данные.

---

## 6. Карточка подразделения

Карточка подразделения содержит:

* название подразделения;
* ФИО руководителя;
* должность руководителя;
* численность подразделения;
* административного ассистента рядом с директором/руководителем, если он найден в выбранной ветке и привязан к верхнему подразделению текущей диаграммы.

Если название отсутствует, используется:

```text
Без названия
```

Если руководитель отсутствует, используется:

```text
Нет руководителя
```

Для отображения должности используется сокращенная версия должности.

Если в названии должности присутствует символ `/`, отображается только часть строки до него.

Численность отображается следующим образом:

* при включенном показе вакансий используется `totalWithVacancies`;
* при скрытых вакансиях используется `staffCount`.

В режиме сценарного моделирования карточка подразделения может отображать метку состояния:

* `NEW` — новое подразделение;
* `Изменен` — измененное подразделение;
* `Перемещен` — перемещенное подразделение;
* `Удален` — удаленное подразделение, если используется мягкое удаление.

Для подразделения в режиме моделирования доступны действия:

* добавить подразделение;
* добавить сотрудника;
* добавить вакансию;
* редактировать подразделение;
* переместить подразделение;
* удалить подразделение.

---

## 7. Сотрудники

Сотрудники отображаются только в организационной диаграмме.

Карточка сотрудника содержит:

* ФИО;
* должность;
* проект или список проектов, если данные есть в API или были добавлены в сценарии.

Если ФИО отсутствует, используется:

```text
Сотрудник
```

Сотрудник отображается только если значение поля `count` после преобразования в число больше либо равно `1`.

Поддерживаются значения:

```text
1
1.0
1,0
```

Руководитель подразделения исключается из списка отображаемых сотрудников для предотвращения дублирования.

Административный ассистент также может исключаться из нижнего общего списка сотрудников, если он уже отображен отдельной карточкой рядом с директором/руководителем верхней структуры выбранной диаграммы.

Сотрудник считается руководителем, если:

* совпадает идентификатор сотрудника и руководителя;
* либо совпадает ФИО сотрудника и руководителя без учета регистра.

В режиме сценарного моделирования сотрудник может быть:

* добавлен;
* отредактирован;
* удален;
* перемещен в другое подразделение.

Для нового сотрудника генерируется локальный идентификатор.

Карточка сотрудника может отображать метку состояния:

* `NEW` — новый сотрудник;
* `Изменен` — измененный сотрудник;
* `Перемещен` — перемещенный сотрудник;
* `Удален` — удаленный сотрудник, если используется мягкое удаление.

Для сотрудника в режиме моделирования доступны действия:

* редактировать;
* переместить;
* удалить.

При перемещении сотрудника пользователь выбирает новое подразделение.

После перемещения:

* сотрудник исключается из старого подразделения;
* добавляется в новое подразделение;
* пересчитывается численность старой и новой ветки;
* изменение фиксируется в сценарии.

---

## 8. Вакансии

Вакансии отображаются только в организационной диаграмме и только при включенном режиме:

```text
Показывать вакансии
```

Карточка вакансии содержит:

* текст `Вакансия`;
* должность вакансии;
* проект, если он указан в API или добавлен в сценарии.

Для вакансий также используется сокращенная версия должности.

Вакансии не открывают детальную карточку сотрудника.

В режиме сценарного моделирования вакансия может быть:

* добавлена;
* отредактирована;
* удалена;
* перемещена в другое подразделение.

Для новой вакансии генерируется локальный идентификатор.

Карточка вакансии может отображать метку состояния:

* `NEW` — новая вакансия;
* `Изменен` — измененная вакансия;
* `Перемещен` — перемещенная вакансия;
* `Удален` — удаленная вакансия, если используется мягкое удаление.

Для вакансии в режиме моделирования доступны действия:

* редактировать;
* переместить;
* удалить.

При перемещении вакансии пользователь выбирает новое подразделение.

После перемещения:

* вакансия исключается из старого подразделения;
* добавляется в новое подразделение;
* пересчитывается численность старой и новой ветки;
* изменение фиксируется в сценарии.

---

## 9. Переключатель вакансий

По умолчанию вакансии отображаются.

При включенном режиме:

* вакансии отображаются в диаграмме;
* численность подразделений отображается с учетом вакансий;
* используется показатель `totalWithVacancies`;
* в классическом дизайне численность подсвечивается синим цветом.

При выключенном режиме:

* вакансии скрываются;
* отображается только фактическая численность сотрудников;
* используется показатель `staffCount`.

Переключатель вакансий работает во всех режимах отображения:

* текущая структура;
* целевая структура;
* изменения.

При выключенном переключателе вакансии не отображаются в диаграмме, но могут оставаться частью сценария моделирования.

---

## 10. Расчет численности

Для каждого подразделения рассчитываются:

```text
staffCount
vacancyCount
totalWithVacancies
```

`staffCount` включает сотрудников текущего подразделения и всех дочерних подразделений.

`vacancyCount` включает вакансии текущего подразделения и всех дочерних подразделений.

`totalWithVacancies` рассчитывается как:

```text
totalWithVacancies = staffCount + vacancyCount
```

Особенности расчета:

* руководитель подразделения участвует в расчете численности;
* из визуального списка сотрудников руководитель исключается только для предотвращения дублирования в диаграмме;
* показатели рассчитываются рекурсивно по всей подчиненной структуре;
* в левой панели показатели численности не отображаются.

В режиме сценарного моделирования пересчет выполняется после каждой операции:

* добавление подразделения;
* редактирование подразделения;
* удаление подразделения;
* перемещение подразделения;
* добавление сотрудника;
* редактирование сотрудника;
* удаление сотрудника;
* перемещение сотрудника;
* добавление вакансии;
* редактирование вакансии;
* удаление вакансии;
* перемещение вакансии.

Пересчет применяется к рабочей структуре сценария TO BE.

Исходная структура AS IS не изменяется.

В аналитической панели отображаются итоговые показатели целевой структуры и отклонения от текущей структуры:

```text
Сотрудники: 742 (+8)
Вакансии: 61 (-3)
Всего: 803 (+5)
Подразделения: 48 (+1)
```

## 11. Сортировка сотрудников и вакансий

Перед отображением выполняется сортировка.

Сначала отображаются сотрудники.

После сотрудников отображаются вакансии.

Основной порядок сотрудников определяется полем:

```text
sub_level
```

Чем меньше значение, тем выше позиция сотрудника в списке.

Если `sub_level` отсутствует, используется резервная сортировка по ключевым словам в должности.

| Ключевое слово | Приоритет |
| -------------- | --------: |
| директор       |         1 |
| руководитель   |         2 |
| начальник      |         3 |
| лидер          |         4 |
| lead           |         4 |
| ведущий        |         5 |
| старший        |         6 |
| главный        |         6 |

Для остальных должностей используется приоритет:

```text
100
```

Если приоритет совпадает, выполняется сортировка по ФИО либо должности с использованием русской локали.

При добавлении, редактировании или перемещении сотрудника или вакансии сортировка выполняется повторно.

Для новых сотрудников и вакансий может быть задано значение `sub_level`.

Если `sub_level` не задан, используется резервная сортировка по должности.

## 12. Детальная карточка сотрудника

Карточка открывается по клику на сотрудника в организационной диаграмме.

Карточка также открывается по клику на административного ассистента, если он отображается отдельной карточкой рядом с директором/руководителем.

Карточка не открывается:

* для подразделений;
* для вакансий.

Карточка может содержать:

* фотографию;
* ФИО;
* должность;
* проект;
* телефон;
* email;
* тип занятости;
* статус.

Поля отображаются только при наличии данных.

Если фотография отсутствует, отображается заглушка.

Email отображается как ссылка:

```text
mailto:
```

Карточка закрывается:

* кнопкой `×`;
* кликом по затемненной области вне карточки.

В режиме моделирования карточка сотрудника открывается для сотрудников, включая новых сотрудников, добавленных локально в сценарии.

Контекстное меню сотрудника не должно мешать открытию детальной карточки.

Если пользователь нажимает кнопку контекстного меню `⋮`, открывается меню действий.

Если пользователь нажимает на саму карточку сотрудника, открывается детальная карточка.

---

## 13. Поддерживаемые поля сотрудника

### Проект

Определяется из одного из полей:

```text
projects
project
project_name
projectName
project_title
```

### Телефон

Определяется из одного из полей:

```text
phone
phone_number
mobile
work_phone
```

### Фотография

Определяется из одного из полей:

```text
photo
photo_url
avatar
avatar_url
```

### Тип занятости

Используется поле:

```text
type_employment
```

Во внутренней структуре сценария поле может использоваться как:

```text
typeEmployment
```

### Статус

Используется поле:

```text
state
```

### Идентификатор сотрудника

Используется поле:

```text
id
```

Для новых сотрудников генерируется локальный идентификатор.

### Уровень должности

Используется поле:

```text
sub_level
```

Во внутренней структуре сценария поле может использоваться как:

```text
subLevel
```

### Административный ассистент

Сотрудник считается административным ассистентом, если его должность содержит строку:

```text
административный ассистент
```

Проверка выполняется без учета регистра.

Для определения используется сокращенная или исходная должность сотрудника:

```text
rawPosition
position
```

Если административный ассистент найден в отображаемой ветке, он должен быть выведен рядом с директором/руководителем верхней карточки текущей диаграммы.

### Дополнительные поля сценария

Для объектов, измененных в рамках сценария, используется служебное поле:

```text
scenarioState
```

Возможные значения:

```text
added
changed
moved
removed
```

## 14. Сокращение должностей

Для руководителей, сотрудников и вакансий используется сокращенное отображение должности.

Если должность содержит символ `/`, отображается только часть строки до него.

Пример:

```text
Руководитель отдела / Юридический департамент
```

отображается как:

* при добавлении вакансии;

## 16. Экспорт в PDF

Пользователь может экспортировать текущую диаграмму в PDF по кнопке:

```text
📄 Экспорт в PDF
```

Экспортируется только текущая отображаемая диаграмма.

Во время экспорта:

* кнопка блокируется;
* текст кнопки меняется на `Экспорт...`;
* поверх SVG-узлов временно добавляются рамки;
* диаграмма преобразуется в изображение PNG;
* создается PDF;
* файл сохраняется под именем:

```text
orgchart.pdf
```

Ориентация PDF выбирается автоматически:

* `landscape`, если ширина больше либо равна высоте;
* `portrait`, если высота больше ширины.

Если диаграмма отсутствует, отображается сообщение:

```text
Нет диаграммы для экспорта
```

Если экспорт завершился ошибкой, отображается сообщение об ошибке.

Экспорт работает для текущего режима отображения.

Если выбран режим целевой структуры, в PDF попадает TO BE-структура.

Если выбран режим текущей структуры, в PDF попадает AS IS-структура.

Если выбран режим изменений, в PDF попадает диаграмма изменений.

Если пользователь выбрал конкретное подразделение, экспортируется диаграмма именно выбранного подразделения.

В PDF должны попадать:

* новые подразделения;
* новые сотрудники;
* новые вакансии;
* измененные объекты;
* перемещенные объекты;
* актуальная численность;
* визуальные метки сценарных изменений;
* проекты сотрудников и вакансий;
* административный ассистент рядом с директором/руководителем, если он отображается в текущей диаграмме.

## 18. Адаптивность

При ширине экрана до 700 пикселей:

* интерфейс перестраивается в вертикальную компоновку;
* боковая панель располагается сверху;
* диаграмма располагается снизу;
* высота боковой панели ограничивается;
* область диаграммы занимает оставшуюся часть экрана.

При ширине экрана до 1100 пикселей:

* правая панель изменений скрывается;
* интерфейс использует две основные области: левую панель и диаграмму.

В полной версии интерфейса используется трехколоночная компоновка:

* левая панель навигации и управления;
* центральная область диаграммы;
* правая панель изменений сценария.

12. Исключение административного ассистента из нижнего общего списка сотрудников, если он выведен рядом с директором/руководителем.

14. Отрисовка организационной диаграммы.
15. Открытие карточек сотрудников.
16. Выполнение сценарных операций.
17. Пересчет численности.
18. Обновление диаграммы.
19. Обновление панели изменений.
20. Сравнение AS IS и TO BE.

## 20. Проекты сотрудников и вакансий

В API добавлено поле:

```text
projects
```

Пример:

```json
{
  "id": "18594a1d-236c-11f1-aff4-d85ed308d2c7",
  "full_name": "Кутнях Руслан Николаевич",
  "position": "Руководитель проектов /Дирекция по строительству/, (ЛЕГЕНДА МАГНИТОГОРСКОЙ ООО)",
  "projects": "БС32; СОШ_МС5",
  "sub_level": "3.0",
  "count": "1",
  "type_employment": "Основное место работы",
  "state": "Работает",
  "email": "r.kutnyah@legenda-dom.ru"
}
```

Поле `projects` должно поддерживаться наравне с ранее поддерживаемыми полями проекта:

```text
projects
project
project_name
projectName
project_title
```

Во внутренней структуре приложения значение нормализуется в поле:

```text
project
```

Если в поле указано несколько проектов через точку с запятой, они отображаются в читаемом виде.

Пример:

```text
БС32; СОШ_МС5
```

отображается как:

```text
БС32; СОШ_МС5
```

Проект должен отображаться:

* в карточке сотрудника в организационной диаграмме;
* в карточке вакансии в организационной диаграмме;
* в детальной карточке сотрудника;
* в карточке административного ассистента, если у него указан проект;
* в PDF-экспорте текущей диаграммы.

Проект не должен выводиться в левом дереве подразделений.

Левое дерево сохраняет прежнее назначение и отображает только подразделения.

В режиме сценарного моделирования поле проекта должно поддерживаться:

* при добавлении сотрудника;
* при редактировании сотрудника;
* при добавлении вакансии;
* при редактировании вакансии;
* при перемещении сотрудника или вакансии между подразделениями.

При перемещении сотрудника или вакансии проект сохраняется.

При сбросе сценария проекты возвращаются к значениям исходной структуры AS IS.

## 21. Административный ассистент

Административный ассистент определяется по должности сотрудника.

Сотрудник считается административным ассистентом, если его должность содержит строку:

```text
административный ассистент
```

Проверка выполняется без учета регистра.

Для проверки используются поля:

```text
rawPosition
position
```

Административный ассистент должен отображаться рядом с директором/руководителем верхней карточки текущей диаграммы.

Если пользователь выбрал в левом дереве подразделение, например:

```text
Дирекция по персоналу
```

и внутри этой ветки есть сотрудник с должностью:

```text
Административный ассистент
```

то такой сотрудник должен быть выведен как отдельная карточка рядом с карточкой выбранной дирекции и ее руководителя.

Административный ассистент не должен оставаться внизу диаграммы как обычный сотрудник, если он уже выведен рядом с директором/руководителем.

Поведение:

* ассистент ищется внутри всей отображаемой ветки выбранного подразделения;
* если ассистент найден, он добавляется в плоские данные диаграммы как прямой дочерний элемент верхнего узла выбранной визуализации;
* исходная структура данных API при этом не изменяется;
* расчет численности не меняется;
* ассистент остается сотрудником с точки зрения численности;
* при клике на карточку ассистента открывается детальная карточка сотрудника;
* контекстное меню ассистента работает как контекстное меню сотрудника;
* в режиме AS IS ассистент отображается без возможности изменения;
* в режиме TO BE ассистент может редактироваться, перемещаться или удаляться через сценарные операции сотрудника;
* в режиме изменений ассистент отображается только если по нему есть сценарное изменение или если он связан с отображаемой измененной веткой.

Если в отображаемой ветке найдено несколько административных ассистентов, на первом этапе отображается первый найденный ассистент по порядку обхода дерева.

На следующем этапе допускается расширение логики до отображения нескольких ассистентов.

Карточка административного ассистента содержит:

* метку `Административный ассистент`;
* ФИО;
* должность;
* проект, если указан;
* визуальную метку сценарного состояния, если ассистент был добавлен, изменен, перемещен или удален в сценарии.

Карточка административного ассистента должна попадать в PDF-экспорт текущей диаграммы.

## 22. Соответствие требованиям CR_modeling

Режим сценарного моделирования должен включать полный набор операций над организационной структурой:

* добавление подразделений;
* добавление сотрудников;
* добавление вакансий;
* удаление подразделений;
* удаление сотрудников;
* удаление вакансий;
* редактирование подразделений;
* редактирование сотрудников;
* редактирование вакансий;
* перемещение подразделений;
* перемещение сотрудников;
* перемещение вакансий.

Все операции выполняются только над рабочей структурой сценария TO BE.

Исходная структура AS IS остается неизменной.

Сценарий хранит список операций, а не считается заменой исходных данных API.

На текущем этапе сценарий хранится локально в памяти браузера.

Архитектура клиентской части должна позволять без изменения базового подхода добавить на следующем этапе:

* сохранение сценария в базу данных;
* загрузку сценария из базы данных;
* версионирование сценариев;
* согласование сценариев;
* историю изменений;
* сравнение нескольких сценариев.

Пользовательский интерфейс сценария должен включать:

* название сценария;
* переключатель режимов AS IS / TO BE / Изменения;
* аналитические показатели;
* кнопку сравнения;
* кнопку сброса сценария;
* правую панель изменений;
* контекстное меню на карточках объектов;
* экспорт текущей версии диаграммы в PDF.

Режимы отображения:

* `Текущая структура` показывает исходные данные API без сценарных изменений;
* `Целевая структура` показывает результат применения сценария;
* `Изменения` показывает только измененные объекты и измененные ветки.

После любой сценарной операции выполняется пересчет:

```text
staffCount
vacancyCount
totalWithVacancies
```

Пересчет применяется ко всей рабочей структуре TO BE и должен соответствовать текущей логике расчета приложения.

Визуальные состояния объектов:

* `NEW` — новый объект;
* `Изменен` — измененный объект;
* `Перемещен` — перемещенный объект;
* `Удален` — удаленный объект, если используется мягкое удаление.

PDF-экспорт должен работать для текущего режима отображения и включать:

* актуальную структуру;
* актуальную численность;
* сценарные метки;
* проекты сотрудников и вакансий;
* административного ассистента рядом с директором/руководителем, если он отображается на диаграмме.
