# Задача: реализовать сценарное моделирование организационной структуры (Org Structure Viewer)

## Контекст

Существует приложение **Org Structure Viewer**, которое визуализирует организационную структуру холдинга на основании данных, получаемых из API.

Текущая функциональность:

* загрузка структуры из API;
* отображение дерева подразделений;
* отображение подразделений, сотрудников и вакансий;
* расчет численности;
* карточки сотрудников;
* экспорт организационной диаграммы в PDF.

Необходимо расширить приложение и добавить полноценный режим **сценарного моделирования организационной структуры**.

---

# Цель

Пользователь должен иметь возможность создавать и анализировать целевую организационную структуру (TO BE), не изменяя текущую структуру (AS IS).

Сценарий должен позволять:

* добавлять подразделения;
* добавлять сотрудников;
* добавлять вакансии;
* удалять сотрудников;
* удалять вакансии;
* изменять подразделения;
* перемещать сотрудников между подразделениями;
* перемещать вакансии между подразделениями;
* видеть результат в виде актуальной организационной диаграммы;
* видеть пересчитанную численность;
* экспортировать результат в PDF.

На первом этапе все изменения выполняются локально в памяти браузера.

Сохранение сценариев в БД будет реализовано на следующем этапе, поэтому архитектура должна быть заранее подготовлена к этому.

---

# Основная концепция

Необходимо ввести новую сущность:

```text
Сценарий моделирования
```

Сценарий представляет собой набор изменений относительно текущей организационной структуры.

Пользователь никогда не изменяет исходные данные напрямую.

После загрузки структуры из API создаются:

```text
Текущая структура (AS IS)
```

и

```text
Рабочая структура сценария (TO BE)
```

Все операции выполняются только над рабочей структурой сценария.

Исходная структура остается неизменной.

---

# Архитектура сценария

Сценарий должен хранить изменения, а не полную копию оргструктуры.

Предпочтительный подход:

```javascript
{
  id: "scenario-1",
  name: "Оптимизация Дирекции строительства",
  baseDate: "2026-06-15",

  operations: [
    {
      type: "addDepartment",
      parentId: "dep123"
    },

    {
      type: "addEmployee",
      departmentId: "dep456"
    },

    {
      type: "removeVacancy",
      vacancyId: "vac789"
    }
  ]
}
```

Система должна быть спроектирована так, чтобы на следующем этапе сценарии можно было сохранять и загружать из БД без изменения клиентской архитектуры.

---

# Новый пользовательский интерфейс

Добавить верхнюю панель сценария.

Пример:

```text
────────────────────────────────────
Сценарий моделирования

Сценарий:
Оптимизация Дирекции строительства

Изменений: 12

[ Сбросить сценарий ]
[ Экспорт PDF ]
────────────────────────────────────
```

---

# Режимы отображения

Добавить переключатель:

```text
○ Текущая структура
○ Целевая структура
○ Изменения
```

## Текущая структура

Показывает структуру, полученную из API.

Без изменений.

## Целевая структура

Показывает результат применения сценария.

Именно этот режим используется для моделирования.

## Изменения

Показывает только изменения относительно исходной структуры.

Пример:

```text
+ Отдел BIM
+ 3 сотрудника
- 2 вакансии
⇄ 1 сотрудник переведен
```

---

# Панель изменений

Добавить правую боковую панель.

Пример:

```text
Изменения сценария

+ Отдел BIM

+ Вакансия BIM-координатор

- Вакансия Инженер ПТО

⇄ Иванов И.И.
из Отдела А
в Отдел Б

+ Сидоров С.С.
```

При клике на изменение диаграмма должна автоматически центрироваться на соответствующем объекте.

---

# Контекстное меню объектов

Для всех действий использовать контекстное меню на карточках диаграммы.

Иконка:

```text
⋮
```

---

# Действия для подразделения

Меню подразделения:

```text
Добавить подразделение
Добавить сотрудника
Добавить вакансию

Редактировать подразделение

Переместить подразделение
Удалить подразделение
```

---

# Действия для сотрудника

Меню сотрудника:

```text
Редактировать
Переместить
Удалить
```

---

# Действия для вакансии

Меню вакансии:

```text
Редактировать
Переместить
Удалить
```

---

# Добавление подразделения

Пользователь указывает:

* название подразделения;
* руководителя (необязательно);
* должность руководителя (необязательно).

После сохранения:

* подразделение появляется в дереве;
* подразделение появляется в диаграмме;
* выполняется пересчет численности.

Для новых подразделений генерируется локальный идентификатор.

---

# Добавление сотрудника

Поля формы:

Обязательные:

* ФИО;
* должность.

Необязательные:

* email;
* телефон;
* проект;
* тип занятости;
* статус.

После сохранения сотрудник:

* появляется в подразделении;
* участвует в расчете численности;
* отображается в диаграмме;
* отображается в карточке сотрудника.

Для новых сотрудников генерируется локальный идентификатор.

---

# Добавление вакансии

Поля формы:

Обязательные:

* должность.

Необязательные:

* проект;
* sub_level.

После сохранения вакансия:

* появляется в подразделении;
* участвует в расчете vacancyCount;
* отображается в диаграмме.

---

# Удаление сотрудника

После удаления:

* сотрудник исчезает из структуры;
* пересчитывается staffCount;
* пересчитываются показатели всех родительских подразделений;
* диаграмма обновляется.

Рекомендуется использовать мягкое удаление внутри сценария через операцию:

```javascript
{
  type: "removeEmployee"
}
```

а не физическое удаление объекта.

---

# Удаление вакансии

После удаления:

* вакансия исключается из структуры сценария;
* пересчитывается численность;
* диаграмма обновляется.

---

# Перемещение сотрудника

Пользователь выбирает новое подразделение.

После перемещения:

* сотрудник удаляется из старого подразделения;
* добавляется в новое подразделение;
* пересчитывается численность обеих веток;
* изменение фиксируется в сценарии.

---

# Перемещение вакансии

Аналогично перемещению сотрудника.

---

# Перемещение подразделения

Пользователь может изменить родительское подразделение.

После перемещения:

* подразделение переносится в новую ветку;
* сохраняются все вложенные подразделения;
* сохраняются сотрудники;
* сохраняются вакансии;
* пересчитывается численность всей структуры.

---

# Редактирование подразделения

Допускается изменение:

* названия;
* руководителя;
* должности руководителя.

После сохранения изменения сразу отображаются в дереве и диаграмме.

---

# Пересчет численности

После любой операции необходимо автоматически выполнять полный пересчет:

```text
staffCount
vacancyCount
totalWithVacancies
```

Логика расчета должна полностью соответствовать текущей реализации приложения.

Пересчет должен распространяться на все затронутые родительские подразделения.

---

# Визуальное отображение изменений

Все изменения должны быть явно видны пользователю.

## Новый объект

Отображать зеленую метку:

```text
NEW
```

или

```text
Добавлен
```

---

## Измененный объект

Отображать синюю метку:

```text
Изменен
```

---

## Перемещенный объект

Отображать фиолетовую метку:

```text
Перемещен
```

---

## Удаленный объект

Отображать красную метку:

```text
Удален
```

До окончательного подтверждения рекомендуется отображать удаленные объекты зачеркнутыми с возможностью отмены действия.

---

# Аналитическая панель

В верхней части сценария отображать сводные показатели:

```text
Сотрудники: 742 (+8)

Вакансии: 61 (-3)

Всего: 803 (+5)

Подразделения: 48 (+1)
```

Показатели должны автоматически обновляться после любого изменения.

---

# Сравнение AS IS / TO BE

Добавить отдельное окно сравнения.

Пример:

| Показатель    | Было | Стало | Изменение |
| ------------- | ---: | ----: | --------: |
| Подразделения |   48 |    49 |        +1 |
| Сотрудники    |  742 |   750 |        +8 |
| Вакансии      |   61 |    58 |        -3 |
| Всего         |  803 |   808 |        +5 |

Кнопка открытия:

```text
Сравнить
```

---

# Сброс сценария

Добавить кнопку:

```text
Сбросить сценарий
```

После нажатия:

* удаляются все изменения;
* сценарий пересоздается из исходной структуры;
* диаграмма перестраивается;
* показатели возвращаются к исходным значениям.

---

# Экспорт в PDF

Существующий механизм экспорта сохранить.

Экспорт должен выполняться для текущей версии сценария.

В PDF должны попадать:

* новые подразделения;
* новые сотрудники;
* новые вакансии;
* изменения структуры;
* актуальная численность.

---

# Требования к реализации

Не изменять существующий режим просмотра.

Добавить новый слой моделирования поверх существующей архитектуры.

Логику сценариев реализовать как отдельный модуль:

```text
scenario-manager
```

который отвечает за:

* хранение сценария;
* применение изменений;
* отмену изменений;
* пересчет структуры;
* подготовку данных для визуализации.

---

# Результат

Пользователь должен иметь возможность создать сценарий целевой организационной структуры, выполнить организационные изменения, мгновенно увидеть результат на диаграмме, сравнить его с текущим состоянием и экспортировать целевую структуру в PDF.

Архитектура решения должна быть подготовлена к следующему этапу, в котором сценарии будут сохраняться в базе данных, версионироваться и проходить согласование.
